---
id: RT-002
name: Form submission flow
engine: claude
description: Fills and submits a form (e.g. signup / contact), then verifies the success state and watches the console for errors.
---

Exercise a form submission end-to-end. Use the `kensa-browser` skill's verb set.
Use the app's TEST/STAGING environment — never submit real data to production.

1. Preflight: `kensa-cli browser status --format json`. If not reachable, ask the
   user to start Chrome (Tools → Browser → Start) and stop.
2. Open the form page with console capture so JS errors surface:
   `kensa-cli browser open <FORM_URL> --capture-console`.
3. Discover the fields before guessing selectors:
   `kensa-cli browser eval "JSON.stringify([...document.querySelectorAll('input,select,textarea,button')].map(e=>({t:e.tagName,type:e.type,name:e.name,id:e.id})))"`.
4. Fill each field with `kensa-cli browser fill "<selector>" "<value>"` (use clearly
   fake test data). For inputs that need keystroke events, use
   `kensa-cli browser type "<selector>" "<value>" --clear`.
5. Submit: `kensa-cli browser click "<submit selector>" --capture-console`.
6. Verify the success state — wait for the confirmation, e.g.
   `kensa-cli browser wait --text "Thanks" --timeout 10000` or
   `kensa-cli browser wait --selector ".success" --state visible`. Read it back with
   `kensa-cli browser text "<success selector>" --format json`.
7. Screenshot the result:
   `kensa-cli browser screenshot --out .tms/attachments/rt-002-submitted.png --full-page`.
8. Report: did it submit, did validation behave, were there console errors? Negative
   checks count — try an invalid value and confirm the form rejects it. File defect
   cases for anything broken via the `kensa-browser` report-back loop.

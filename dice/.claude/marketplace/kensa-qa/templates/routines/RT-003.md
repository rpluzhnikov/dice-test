---
id: RT-003
name: Visual baseline capture
engine: claude
description: Captures full-page screenshots of key pages at a fixed viewport so later runs can be diffed against the baseline.
---

Capture a visual baseline of the key pages. Use the `kensa-browser` skill's verb set.

1. Preflight: `kensa-cli browser status --format json`. If not reachable, ask the
   user to start Chrome (Tools → Browser → Start) and stop.
2. For each page in the baseline set (ask the user for the list once, e.g. home,
   pricing, dashboard):
   - `kensa-cli browser open <PAGE_URL> --wait networkidle` so async content settles.
   - `kensa-cli browser screenshot --out .tms/attachments/baseline/<page>.png --full-page`.
3. Note the page `title` and `url` for each (`--format json`) alongside the
   screenshot so a later run can confirm it captured the same page.
4. Report the list of baseline files written. On a **later** run, re-capture into a
   separate folder (e.g. `.tms/attachments/current/`) and describe any visible
   differences from the baseline — layout shifts, missing images, broken styling.
   File a defect case for genuine regressions via the `kensa-browser` report-back
   loop; don't flag expected content changes.

> Note: this routine captures and compares by eye/description — it is not a
> pixel-diff tool. For exact pixel diffing, wire a dedicated visual-regression tool;
> this gives you a fast, committable baseline to eyeball against.

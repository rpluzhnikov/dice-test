# Grounding

The single largest source of confident-wrong answers in long reasoning is mistreating speculation as established fact. A thought built on an unchecked premise looks identical to a thought built on a verified one — and the chain that follows inherits confidence neither thought has earned.

This reference is the defense. Six tools, applied unevenly: some on every thought (cheap), some only when stakes are high (expensive but essential).

## The three knowledge classes

Every claim in every thought is one of three things. Knowing which is half the discipline.

**Verified.** Observed directly *in this session*. You read the file, ran the query, checked the metric, ran the test. The claim is grounded in concrete evidence you can point to.

**Believed-but-unverified.** Plausible from prior knowledge, training data, or context — but not actually checked here. It feels true. It usually *is* true. But it hasn't been confirmed for this specific situation.

**Speculative.** Generated by reasoning to fill a gap. Could easily be wrong. The mind produced it because a gap needed filling, not because the world supplied evidence.

The discipline is to mark each claim. Not exhaustively — that's paralysis — but for any non-trivial premise that downstream reasoning will lean on.

```
State:
  - DB query time tripled at 14:00. [VERIFIED — metrics dashboard]
  - The 14:00 deploy touched the payment SDK. [VERIFIED — deploy log]
  - Payment SDK uses sync calls in the new version. [BELIEVED — recall from changelog, not checked in this session]
  - Sync calls hold connections longer. [VERIFIED — fundamental, true in general]
  - The DB pool is sized for the old async pattern. [SPECULATIVE — generated to complete the chain, not checked]

Risk surface: the believed claim and the speculative claim. The whole hypothesis hinges on both being true.

Action before acting: verify both. Read the SDK changelog. Check pool config.
```

The rule of thumb: when the stakes are high and a believed-or-speculative claim is load-bearing, *spend the cheap tool call to upgrade it to verified.* Reasoning five steps deep on top of an unchecked assumption is a false economy.

When the stakes are low and verification is expensive, proceed but flag the uncertainty in the output. "If the SDK is still using sync calls in this version (likely but not verified), then…"

What kills people: not the speculative claims they know are speculative. The *believed* claims that *feel* verified. Those are where confabulation lives.

## The ReAct pattern

ReAct = Reason + Act. Alternate steps of internal reasoning with steps of external observation. Each observation is grounded; each reasoning step builds on the most recent observation rather than running blind for many steps.

```
Reason: "The error is probably in the auth middleware."
Act: Read the auth middleware. Check the recent commits.
Observe: Middleware unchanged in 60 days; no recent commits.
Reason: Update — middleware is unlikely to be the cause. Where else does the request flow?
Act: Trace the call path.
Observe: Request hits a new rate-limiter added last week.
Reason: New hypothesis — rate-limiter is the cause.
Act: Test by disabling rate-limiter in staging.
Observe: Errors disappear.
Reason: Confirmed. Fix the rate-limiter configuration.
```

Contrast with the failure mode — reasoning five steps deep without observing:

```
Reason: "The error is probably in the auth middleware."
Reason: "If it's auth middleware, the issue might be token validation."
Reason: "Token validation could fail if JWT signature is invalid."
Reason: "JWT signature could be invalid if the signing key rotated."
Reason: "Therefore, check the key rotation log."
[All five steps were wrong — the actual cause was a rate-limiter the chain never considered.]
```

Five thoughts of confident reasoning, all wrong, because none of them touched reality.

The rule: when a thought depends on a fact you could check, *check it before building three more thoughts on it.* The cost of one tool call is much less than the cost of unwinding five wrong thoughts.

Counter-rule: don't fetishize verification when reasoning is reasonable and the stakes are low. ReAct is for chains where being wrong would matter. For "what's the syntax for X" you don't need to alternate — just answer.

## Falsifiability

For any non-trivial claim, ask: *what observation, if I made it, would prove this wrong?*

If nothing could prove it wrong, the claim is unfalsifiable — and unfalsifiable claims are not knowledge. They feel like knowledge because they explain everything; they explain everything because they're empty.

Falsifiability is a *test for whether a claim is doing real work.* A claim that real evidence could refute is a claim that real evidence supports. A claim that bends to fit any evidence supports nothing.

### Examples

**Unfalsifiable, sneaks past:**
- "The team is just under-performing." (What would make this false? It's vague enough to bend.)
- "The architecture is wrong." (Wrong how? In what way would right look different?)
- "Users want a better experience." (What user behavior would refute this?)

**Falsifiable, doing real work:**
- "The team's velocity has dropped 30% in the last quarter." (Could be falsified by checking velocity metrics.)
- "The architecture can't scale past 10K RPS." (Could be falsified by load test.)
- "Users abandon the checkout at step 3 more than at any other step." (Could be falsified by funnel data.)

The exercise pays off twice: it forces you to be specific about what you're claiming, and it gives you a concrete test to run when you need to verify.

### As a thought-level check

Before treating a claim as established, ask "what would prove this wrong?" If you can't answer, the claim is impressionistic — useful for direction-setting, not for building downstream conclusions on.

A common debugging anti-pattern: "It's probably a race condition." Race conditions are extremely flexible explanations. They can be made to fit almost any intermittent symptom. That flexibility makes the hypothesis *low-information* — confirming it doesn't tell you much. Either narrow it (what specific race, between which threads, on what shared state?) or treat it as a placeholder rather than a finding.

## Map-territory thinking

You reason about a system using a *model* of it: the architecture diagram, the spec, the mental model of how the code works, the metrics dashboard, the test results, the documentation. These are all *maps*.

The actual running system, the actual user, the actual data are the *territory*.

Maps lose information by design. They're useful precisely because they're simpler than what they represent. But the simplification means they always omit something — and when reasoning fails, it's often because reasoning operated on the map while the bug lived in the territory.

| Map | Common omissions |
|---|---|
| Architecture diagram | Failure modes, race conditions, partial outages, version skew |
| Code spec | Real-world input distributions, performance under load, edge cases |
| Test suite | Untested paths, integration behavior, edge cases not imagined |
| Mental model of the code | Recent changes, what production actually does, the parts you forgot exist |
| Metrics dashboard | What the metrics don't measure, sampling bias, aggregation hiding tails |
| Documentation | What was true when written, vs. now |

### Signs the map is wrong

- Expected behavior repeatedly fails to match observed behavior.
- The "obvious" answer based on the diagram doesn't actually work.
- Multiple people insist the system works one way, but logs show otherwise.
- A test passes but the bug is still happening.

When predictions repeatedly disagree with observations: *suspect the map first, not the territory.* The territory is what it is. The map is whatever you (or someone) drew.

### Practical drill

When a claim is map-based, name it as such:

```
"According to the architecture diagram, the cache sits between the API and the DB."
                                       ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                                       This is map. Has anyone verified this is true in production?
```

In a complex system, the diagram is often weeks-to-years out of date. The code is the territory of the design; the running system is the territory of the code. Both can diverge from what's documented.

### Special case: tests

Tests are a map of expected behavior. Green tests mean "the system behaves as the tests expect" — not "the system is correct." If the test suite doesn't cover the path where the bug lives, green tests provide false reassurance. Many production bugs live in the gap between "what the tests check" and "what users actually do."

## Scientific method as a thought structure

For any non-trivial hypothesis, the scientific method gives a four-step structure:

1. **Hypothesis.** State the candidate explanation clearly. Make it specific enough that it could be wrong.
2. **Prediction.** If this hypothesis is true, what observation would I expect? If it's false, what would I expect instead? The prediction must *differentiate* — if both hypothesis and its negation predict the same observation, the observation tells you nothing.
3. **Test.** Make the observation. Run the experiment. Check the data.
4. **Update.** Was the prediction borne out? If yes, the hypothesis survives — note this isn't proof, just non-refutation. If no, the hypothesis is wrong; generate a new candidate.

### Where the method protects you

It forces specificity. "The cache is the problem" is not a hypothesis the method can work with. "Disabling the cache will drop p99 latency by at least 50%" is.

It builds in falsifiability. The prediction step is the falsifiability test — a hypothesis without a differentiating prediction is not yet scientific.

It enforces the update. People generate hypotheses easily and run tests easily; the discipline that fails is *updating beliefs based on test results*. Tests that disconfirm your hypothesis are valuable — they save you from acting on a wrong theory. Don't reflexively reinterpret a disconfirming result to keep the hypothesis alive.

### A/B testing as scientific method

When you're running an A/B test or any controlled experiment, the structure is identical:

```
Hypothesis: New checkout flow increases conversion by ≥5%.
Prediction: Treatment group shows ≥5% higher conversion than control.
Test: Run for two weeks with 50/50 split, sufficient sample size.
Update: 
  - Result: +2% with p=0.3 → hypothesis not supported. (Not "we need more data" by default — that's a different decision.)
  - Result: +7% with p=0.01 → hypothesis supported.
  - Result: -3% with p=0.01 → hypothesis falsified; treatment is worse.
```

### Debugging as scientific method

Same structure:

```
Hypothesis: The 500s come from connection pool exhaustion.
Prediction: When error rate spikes, pool utilization should be near 100%.
Test: Correlate pool metrics with error spike timeline.
Update: If utilization was 40% during the spike, hypothesis is falsified — look elsewhere.
```

## Calibrated confidence

False precision is worse than honest uncertainty. Stating "I'm 87% confident" when you have no calibration basis adds zero information and creates false-rigor.

Use language that reflects actual confidence, anchored to observable signals.

| Phrase | Meaning |
|---|---|
| "Verified — I just checked." | Highest. Direct observation in this session. |
| "Strong evidence; multiple independent signals agree." | High. Multiple paths converging. |
| "Consistent with the evidence, but other explanations also fit." | Medium. Best current guess, not a finding. |
| "Plausible but not yet supported by anything specific." | Low. Worth testing, not yet worth acting on. |
| "Speculation to keep the chain moving — flag for verification." | Very low. Placeholder. |

Two reliable signals are stronger than one. The strongest claims are ones supported by independent evidence — not the same evidence interpreted multiple ways, but truly separate observations pointing at the same answer. This is the underlying mechanic of multi-approach synthesis (see the dedicated reference).

### The likelihood-ratio intuition

When evidence updates your belief, ask: how much *more likely* is this evidence under the current hypothesis than under its alternatives?

- *Strong evidence:* the observation would be common if hypothesis is true, rare if it's false. (E.g., the bug only happens in US-East and only US-East was affected by the deploy — strongly favors "deploy caused it.")
- *Weak evidence:* the observation is roughly as likely under either. (E.g., "errors went up after deploy" — but errors also go up at peak load. Not very discriminating.)
- *Noise:* equally likely either way. Don't update at all.

You don't need numerical Bayes for this. The qualitative version — "does this observation discriminate between my candidates?" — is what matters. Evidence that fits everything fits nothing.

## Spotting confabulation in your own thoughts

The reason confabulation is dangerous is that it doesn't feel different from real knowing. Both feel like things you know.

Some signals that a thought might be confabulating:

**Suspicious fluency.** You produced a confident answer in a domain where you don't have strong grounding, and you can't immediately point to the source of that confidence. Fluent reasoning on shaky ground is the natural failure mode of LLMs — including this one.

**Specific details with no anchor.** Numbers, dates, version strings, technical specifics that came out fully-formed in a thought, without a source. If you can't recall *where* you know this from, treat it as believed-not-verified.

**Pattern-matched answers.** "This is just like X, so the answer is Y." Useful starting point, but verify — pattern matching is exactly how confabulation feels.

**Conclusions that feel inevitable.** Real reasoning produces conclusions that *could have gone another way given different evidence.* If the conclusion feels like it had to come out this way, the chain may have been guided by what felt right rather than what the evidence supports.

**Premises that "obviously" hold.** Any time you find yourself thinking "of course X is true," that's the moment to ask: *is X actually verified, or just believed?* Strong feelings of obviousness are a feature of confabulation, not a guarantee against it.

### What to do

When you spot the signal, *don't* push past it. The right move is one of:

1. *Verify the suspect claim* before continuing. (Cheap if a tool call can confirm.)
2. *Flag the uncertainty* in the conclusion. ("If X holds — likely but unverified — then Y.")
3. *Branch* to consider the alternative where the suspect claim doesn't hold. (Costs one branch, saves a chain of wrong conclusions if the suspect claim was indeed wrong.)
4. *Stop and tell the user* you'd need to check something before answering confidently.

The instinct to push past these signals to maintain forward momentum is the precise instinct that produces confident-wrong outputs. The discipline is to honor the signal even when it's annoying. (For the in-chain patterns these signals correspond to, see `anti-patterns.md` — particularly "Speculation drift," "Confirmation drift," and "Sunk thoughts.")
